# Images

## Naming convention

Each image file in `content/` follows a stem + optional variant suffix + extension pattern. The base file is the full-resolution original; variant files are siblings named by appending a suffix before the extension.

| Suffix | Used by | Description |
|--------|---------|-------------|
| *(none)* | Source of truth | Full-resolution original. Used as the fallback if no variant exists. |
| `-web` | Web HTML, digital PDF (web-pdf), EPUB | Compressed/smaller version for digital delivery. Usually `.jpg` at reduced dimensions. |
| `-print` | Print interiors (7x10-color, 8.5x11), digital PDF (web-pdf) | Print-optimized variant. Color version for color formats. |
| `-print-bw` | (not used by engine variant system) | Manually created print B&W variant for some photos. |
| `-light` | Digital PDF (web-pdf), EPUB, print interiors | Light-background version of dark-themed diagrams. Dark originals are authored for the web dark theme; `-light.*` is the paper-white version. |
| `-bw` | B&W print interiors (7x10-bw, 6x9) | True grayscale. Auto-generated by the build on demand via Pillow if missing. |

**Examples:**

```
berkeley-smibert.jpg            -- original (full res)
berkeley-smibert-web.jpg        -- web + EPUB variant
berkeley-smibert-print.jpg      -- print color variant
berkeley-smibert-print-bw.jpg   -- print B&W variant (manual; not in auto-swap system)

eternal-thought-diagram.png     -- dark-themed diagram (web HTML)
eternal-thought-diagram-light.png  -- light version (EPUB/PDF)
```

## How variant selection works

Variant selection is hardcoded in the engine -- there is no config key that drives it in Phase 1.

### Web HTML (`--web`)

The `build-book.sh` web loop scans for each base image's web variant explicitly:

```
stem + "-web.jpg"   (preferred)
stem + "-web.jpeg"
stem + "-web.png"
```

If a web variant exists, it's copied to `output/images/` and `src` in the chapter HTML is rewritten to `/book-content/images/{web_variant}?v={filesize}`. The filesize is appended as a cache-buster. If no web variant exists, the base image is used.

### Digital PDF and EPUB (markdown/combined, `_swap_to_web_variants()`)

`build-book-pdf.py::_swap_to_web_variants()` rewrites `src="foo.png"` in the combined markdown/HTML:

1. If `src` is already a variant (`-web.`, `-print.`, `-light.`), leave it alone.
2. Otherwise, look for (in order): `{stem}-web.jpg`, `{stem}-web.jpeg`, `{stem}-web.png`, `{stem}-light.png`, `{stem}-light.jpg`.
3. First match wins. If none, leave the original.

The `-light.*` fallback is intentional: diagrams authored dark for the web theme get a light background for the paper-white digital PDF and EPUB, even when there's no `-web` variant.

### Print interiors (color; `_swap_to_print_variants()`)

`build-book-pdf.py::_swap_to_print_variants()` rewrites `src="foo.png"` in the combined markdown/HTML:

1. If `src` is already a `-print.` or `-light.` variant, leave it alone.
2. Otherwise, look for: `{stem}-print.jpg`, `{stem}-light.png`, `{stem}-light.jpg`.
3. First match wins. If none, leave the original.

After variant selection, every image path is routed through `_print_sized()`, which looks for a pre-baked right-sized version in `content/print-sized/`. If one exists, it's used instead of the full-resolution original; this keeps the build fast and prevents weasyprint from embedding multi-megabyte originals.

### Print interiors (B&W; `_grayscale_images()`)

For `--7x10bw` and `--6x9`, after print-variant selection, `_grayscale_images()` swaps every `<img>` for a `-bw` sibling:

1. Check if `{root}-bw{ext}` exists and is newer than the source.
2. If not, generate it: open the source, convert to grayscale (`ImageOps.autocontrast` with 1% cutoff), save as `-bw{ext}`.

The `-bw` files are gitignored (generated on demand). The `-print-bw.*` files are manual variants that exist for some photos; they're NOT in the engine's automatic `-bw` generation path -- they're just files that happen to be present in `content/`.

### EPUB (`epub_enhance.py`)

The EPUB uses the same combined-markdown image rewriting as the digital PDF (web variants / light fallback). `add_print_diagram_images()` in `epub_enhance.py` also copies any diagram image files (the ones swapped from CSS) directly into the EPUB zip.

### CSS diagram swap (`diagram_images.py`)

Four diagrams are authored as pure CSS in the web HTML (eternal-thought, god-thinks/cascades, lineage, sentence breakdown). For PDF and EPUB, these CSS visualizations are swapped for pre-rendered image plates:

- `diagram_images.swap_print_diagrams(html)` scans the HTML for the CSS diagram containers and replaces them with `<img>` tags pointing at PNG files in `content/`.
- The PNG files themselves are named with a `-light.png` or `-print.png` convention and authored at the trim's resolution.
- For EPUB, `add_print_diagram_images()` copies the referenced files into the EPUB's media folder and adds them to the manifest.

## Image size and Git

**Only build-referenced images are committed.** The original Mind working tree had ~308 image files; many were unused variants, font mockups, or image candidates. The Phase-1 build carried all images over into `content/` without pruning (byte-identical constraint). Phase 2 (via `tools/image_usage_scan.py`, once built) will identify the exact set the build touches and prune the rest.

Cover PSDs (design source files, 1.3 GB) are excluded from the repo entirely and live only in local backups. Built PDFs and EPUBs are also gitignored.

**Auto-generated files** that the build writes to `content/` are gitignored:
- `content/print-sized/` -- prebaked right-sized images for print
- `content/*-bw.*` -- auto-generated grayscale variants
